# Remote Module Spec (網路通訊層)

## 1. 模組定位與職責 (Module Boundary)

`pkg/remote` 是整個 Actor Cluster 的 **gRPC transport adapter**。
它只負責三件事：

1. **Protocol Contract**：定義 Actor Cluster 對外溝通的 protobuf，包括 request envelope、batch、result 與 gRPC stream service。
2. **Node Stream Server**：在 Actor Node 端實作 `ActorService.StreamMessages` bidirectional stream server，接收對端送入的 `BatchRequest`。
3. **Callback Adapter**：每收到一個 `BatchRequest` 就回呼 Node runtime 注入的 `BatchHandler`，並提供 `BatchSender` 作為該 stream 的送出出口。

**職責內 (In Scope)：**
- gRPC protobuf 定義與傳輸協定。
- `pb` 生成後的 Go package layout。
- Actor Node 端 `ActorService.StreamMessages` server 實作。
- Stream `Recv` / `Send` 錯誤轉換。
- `BatchHandler` callback contract。
- `BatchSender.SendBatchResponse` 的 stream send 序列化。
- 提供 `cmd/node` 可掛載的 gRPC stream server 建構方式。

**職責外 (Out of Scope)：**
- 不負責槽位計算與拓樸來源；那是 `pkg/discovery` 的責任。
- 不負責 Actor 業務狀態機與持久化；那是 `internal/node`、`pkg/actor` 與 `pkg/persistence` 的責任。
- 不負責 ownership / op_code / payload / deadline 等 Actor runtime validation；這些屬於 Node runtime。
- 不負責 batch 拆分、同 Actor 保序、worker pool、response batching policy。
- 不規範 stream 對端如何回覆 HTTP / TCP / 外部客戶端。
- 不在第一版實作任何 stream client、conn pool、pending future map、retry policy。
- 不負責 Node-to-Node 轉發；任何送錯 Node 的請求一律由 Node runtime 拒絕，不做內部補救轉送。

## 2. 網路拓樸與設計精神 (Networking Philosophy)

### 2.1 Node-Scoped Batch Invariant
`pkg/remote` 的 server 假設每個 `BatchRequest` 都是送給當前 Actor Node 的 node-scoped batch。
也就是說，單一 gRPC stream 上的單一 batch 只應包含當前 Node 可處理的資料。

```text
BatchRequest to node-A:
  envelopes[0] -> SlotOf(t1, u1) belongs to node-A
  envelopes[1] -> SlotOf(t2, u9) belongs to node-A
  envelopes[2] -> SlotOf(t1, u7) belongs to node-A

BatchRequest must not contain:
  envelopes[i] -> belongs to node-B
```

這個 invariant 帶來三個好處：
- Node runtime 不需要做二次路由或跨 Node 轉發，只要做 ownership fast-check。
- Batch 進入 Node 後可以直接 validate + dispatch，避免在 `pkg/remote` 內做重新分組。
- HTTP/2 stream 的 flow-control 與 backpressure 可以自然歸因到當前 Node stream。

### 2.2 No Internal Hops
Cluster 內部 Node 之間**絕不**代轉業務請求。
一旦 Node runtime 發現封包中的 `(tenant_id, uid)` 不屬於自己當前持有的 slot 範圍，必須立刻回覆可機械判定的錯誤碼，例如 `ERR_WRONG_NODE`。

這個限制是故意的：
- 消除 Multi-hop 帶來的額外延遲與 timeout 疊加。
- 避免 Node-to-Node 轉送時隱性形成第二套 cluster protocol。
- 發生拓樸切換時，Node runtime 僅回覆明確錯誤，不自行修正或轉送。

### 2.3 Remote 只做 I/O，Actor 保持單執行緒純度
網路邊界允許為了 I/O 使用 goroutine、`chan`、mutex。
但這些並發工具不能滲入 Actor core。
`pkg/remote` 只把整批 `BatchRequest` 交給 `BatchHandler`；如何 dispatch 各個 `*pb.RemoteEnvelope` 到對應 Actor，由 Node runtime 決定。

### 2.4 Stream Transport Semantics
`pkg/remote` 只定義 Node 端 transport contract：

```text
stream peer
  -> send BatchRequest

Actor Node
  -> remote.Server receives BatchRequest
  -> remote.Server invokes BatchHandler callback
  -> Node runtime validates / dispatches / aggregates responses
  -> Node runtime calls BatchSender.SendBatchResponse
  -> remote.Server emits BatchResponse on the same bidi stream
```

這裡的「stream」不代表 `pkg/remote` 要理解業務完成語義。
Actor 何時完成、何時合併回應、何時呼叫 `SendBatchResponse`，都由 Node runtime 決定。

## 3. 傳輸協定 (Protocol Contract)

### 3.1 設計原則
- **所有 Actor 請求必帶 `uid`**：這是最小存活單位。
- **必帶 `tenant_id`**：因為持久化分區鍵是 `(tenant_id, uid)`，不能只靠 `uid`。
- **Batch 不帶 `target_node` 欄位**：目標 Node 是 stream 連線本身的外部事實，不塞進 protobuf，避免每包重複攜帶相同資訊。
- **`tx_id` 只代表業務冪等鍵，不拿來當 transport callback key**。
- **回應必須可一對一對應到請求**，因此 transport 層需要獨立的 `request_id`。
- **`request_id` 必須全域唯一**：由上游入口生成，不能只是單一 stream-local counter；不同 Gateway instance、不同連線與不同重連生命週期之間都不得重複。`internal/node` 依賴此 invariant，使用 `(tenant_id, uid, request_id)` 將結果路由回原始 stream。

### 3.2 Protobuf 定義

```protobuf
syntax = "proto3";

package remote;
option go_package = "github.com/frankieli/actor_cluster/pkg/remote/pb";

message RemoteEnvelope {
  int32  tenant_id  = 1;
  int64  uid        = 2;
  string tx_id      = 3;
  uint64 request_id = 4; // transport correlation key, generated by stream peer; must be cluster-global unique
  int32  op_code    = 5;
  bytes  payload    = 6;
  int64  deadline_unix_ms = 7; // optional soft deadline for node-runtime fast reject
}

message RemoteResult {
  int32  tenant_id  = 1; // mirrors RemoteEnvelope for result attribution / audit
  int64  uid        = 2;
  string tx_id      = 3;
  uint64 request_id = 4; // transport correlation key; mirrors the globally unique request_id from request
  int32  op_code    = 5;
  bool   success    = 6;
  string error_code = 7;
  string error_msg  = 8;
  bytes  payload    = 9; // optional business response payload
}

message BatchRequest {
  repeated RemoteEnvelope envelopes = 1;
}

message BatchResponse {
  repeated RemoteResult results = 1;
}

service ActorService {
  rpc StreamMessages(stream BatchRequest) returns (stream BatchResponse);
}
```

### 3.3 為何不用 Unary
高頻場景下，Unary request 代表每一筆訊息都要完整走一次 HTTP/2 header decode、stream 建立、flow-control bookkeeping。
`Stream + Batch` 的目標是把固定開銷攤提到一批訊息上，讓單筆封包的邊際成本更接近 protobuf encode/decode 本身。

## 4. Package Layout 與實作邊界

### 4.1 目錄規劃

```text
pkg/remote/
  proto/
    actor_cluster.proto       # protobuf source of truth
  pb/
    actor_cluster.pb.go       # protoc-gen-go output
    actor_cluster_grpc.pb.go  # protoc-gen-go-grpc output
  server.go                   # ActorService transport adapter implementation
  types.go                    # BatchHandler, BatchSender Go contract
  errors.go                   # stable remote error code constants

cmd/node/
  main.go                     # 組裝 Node runtime handler、remote.Server、grpc.Server
```

`pkg/remote/proto` 是協定源頭。
`pkg/remote/pb` 是生成物，只允許由 `protoc` 更新，不手刻。
`pkg/remote` 根目錄只提供 Node 端 transport adapter 與必要介面，不實作 stream client。

### 4.2 Server 建構介面

```go
package remote

type ServerConfig struct {
    MaxBatchSize int
}

type Server struct {
    pb.UnimplementedActorServiceServer
    handler BatchHandler
    cfg     ServerConfig
}

func NewServer(handler BatchHandler, cfg ServerConfig) *Server
```

`cmd/node` 只需要：

```go
srv := remote.NewServer(nodeHandler, cfg)
grpcServer := grpc.NewServer()
pb.RegisterActorServiceServer(grpcServer, srv)
```

**`MaxBatchSize` 實際值**：

- 函式庫層：若 `cfg.MaxBatchSize <= 0`，`NewServer` 以 **`pkg/remote` 的預設 256** 取代（見 `defaultMaxBatchSize`）。
- 節點二進位：`cmd/node/main.go` 顯式傳入 **`MaxBatchSize: 5000`**，與大批次壓測情境對齊；設計文檔驗收 batch size 掃描（1 / 8 / 64 / 256 等）仍以節點**實際**上限 ≤ `MaxBatchSize` 為前提。

## 5. Server 端設計 (Actor Node Side)

### 5.1 核心介面

```go
package remote

import (
    "context"

    "github.com/frankieli/actor_cluster/pkg/remote/pb"
)

type BatchHandler interface {
    HandleBatch(ctx context.Context, batch *pb.BatchRequest, sender BatchSender) error
}

type BatchSender interface {
    SendBatchResponse(resp *pb.BatchResponse) error
}
```

`BatchHandler` 背後通常是 Node runtime。
它負責：
- 驗證 request envelope。
- 確認本機是否仍持有該 `(tenant_id, uid)` 的 slot。
- 找到或建立對應 Actor。
- 決定同 Actor 保序與不同 Actor 並行策略。
- 聚合 Actor response，並依時間或數量門檻呼叫 `BatchSender.SendBatchResponse`。

### 5.2 StreamMessages 處理流程

```text
for {
  batch := stream.Recv()
  if transport invalid -> return gRPC status error
  handler.HandleBatch(ctx, batch, sender)
}
```

### 5.3 Remote 端驗證項目
`pkg/remote` 僅驗 transport-level guard：

1. `batch != nil`
2. `len(batch.envelopes) <= MaxBatchSize`

每筆 `RemoteEnvelope` 的語義驗證交由 Node runtime，至少包含：

1. `tenant_id > 0`
2. `uid > 0`
3. `tx_id != ""`
4. `request_id != 0`
5. `op_code` 是否在本 Node 支援範圍
6. `payload` 長度是否超過硬上限
7. `(tenant_id, uid)` 是否屬於本 Node 當前 slot 範圍；若任一 envelope 不屬於本 Node，該 envelope 回 `ERR_WRONG_NODE`
8. 若 `deadline_unix_ms` 已過期，直接快速拒絕

### 5.4 Dispatch 併發模型
Dispatch 併發模型不屬於 `pkg/remote`。
Node runtime 可以在 handler 內逐筆同步 dispatch，也可以把 batch 拆成工作項交給固定大小 worker pool。
選擇原則是：

- **不能為每筆 request 開新 goroutine**。
- 若 dispatch 本身主要成本是等待 Actor 回覆，那使用固定大小 worker pool 可以避免單一 stream goroutine 被慢 UID 卡住太久。
- 若同一 batch 很可能含有大量不同 UID，worker pool 應只負責把訊息送進對應 Actor 並等待結果，不得在 pool 裡做業務計算。
- 同一 `(tenant_id, uid)` 必須保序；若採 worker pool，應使用 hash-sharded queue。

### 5.5 Response 順序
`BatchResponse.results` 不要求與 `BatchRequest.envelopes` 保持原始順序，只要求每筆結果帶上 `request_id`。
這樣 Node runtime 能更自由地平行等待不同 Actor 的完成時間，不必為了維持順序額外做排序或 hold 住整批回應。

## 6. 記憶體與效能設計 (Performance & Allocation)

### 6.1 Payload 處理
- Protobuf decode 後的 `payload []byte` 由 `BatchHandler` 決定如何傳給 Actor runtime。
- remote 層不對 payload 做 JSON decode 或反射轉換。
- 若業務層需要進一步 decode，由 `internal/node` 內的業務 Actor / op_code handler 自己決定何時解析。

### 6.2 Send 序列化
`BatchSender.SendBatchResponse` 必須序列化同一條 stream 上的 `Send` 呼叫。
Node runtime 可以從多個 worker / aggregator 呼叫 sender，但 `pkg/remote` 必須確保 gRPC stream 不被多 goroutine 同時寫入。

### 6.3 Hot path 禁止事項
- 不在每筆 request 上做 `time.NewTimer`。
- 不在 remote 核心層使用 `reflect` 或泛型序列化器。
- 若 Node runtime 採 worker pool，不為每筆 request 臨時開 goroutine。

## 7. 錯誤模型 (Error Model)

穩定字串**一律**為大寫 SNAKE、`ERR_` 前綴；**單一真相**在 `pkg/remote/errors.go` 的 `Err*` 常數（字面值與下表一致）。同一字串可用於（1）gRPC `status` 的 message，或（2）`RemoteResult.error_code`。

### 7.1 gRPC transport（`pkg/remote.Server` 以 `status` 回傳）

| 字串 | 語意 |
|------|------|
| `ERR_BAD_REQUEST` | nil batch/response 或不可解析之請求 |
| `ERR_TRANSPORT_CLOSED` | stream Recv/Send 失敗、EOF，或 context 取消 |
| `ERR_BATCH_TOO_LARGE` | `len(envelopes)` 超過 `ServerConfig.MaxBatchSize` |
| `ERR_DEADLINE_EXCEEDED` | handler 回傳 `context.DeadlineExceeded` |
| `ERR_INTERNAL` | handler 回傳非 gRPC status 的未知錯誤 |

### 7.2 `RemoteResult.error_code` — Node / 業務 handler

| 字串 | 語意 |
|------|------|
| `ERR_WRONG_NODE` | 本節點非 `(tenant_id, uid)` 之 owner |
| `ERR_BAD_REQUEST` | 欄位缺失等驗證失敗（與 7.1 同字串值，出現在 result 而非 gRPC） |
| `ERR_ACTOR_UNAVAILABLE` | Actor 無法接受訊息（預留） |
| `ERR_PAYLOAD_TOO_LARGE` | 單筆 envelope payload 超過 Node 上層硬上限 |
| `ERR_REHYDRATION_FAILED` | 自 Cassandra 還原 Actor 狀態失敗 |
| `ERR_INVALID_PAYLOAD` | 業務層：例如 payload 長度/格式不合法 |
| `ERR_INSUFFICIENT_FUNDS` | 業務層：餘額不足 |
| `ERR_PERSISTENCE_FAILED` | 底層 persistence 執行失敗 |

### 7.3 客戶端**未**收到伺服器回應時之合成結果（`cmd/client` 等）

仍寫入 `RemoteResult.error_code`，便於與真正自 Node 回傳之結果統一處理：

| 字串 | 語意 |
|------|------|
| `ERR_DISCOVERY` | `GetNodeIP` 失敗（表未就緒、無 owner 等） |
| `ERR_CONNECTION` | 無法建立 gRPC 或建立 `NodeStreamer` 失敗 |
| `ERR_STREAM_INTERRUPTED` | 串流 `Recv` 錯誤，尚未完成之等待中請求一併失敗 |

### 7.4 與實作對照

- `internal/node`：多 gRPC stream 併發時，回包依 `(tenant_id, uid, request_id)` 綁回正確 stream；見 `05_internal_node_spec.md` 之 `replyRoutes`。
- 新增錯誤碼時：**先**在 `errors.go` 與本節增列，**再**於 handler 使用常數，禁止散佈裸字串。

## 8. Trade-off 分析

### 8.1 Stream + Batch vs Unary
- **選擇 Stream + Batch**：降低每筆 request 的固定 HTTP/2 成本。
- **結論**：Node server 只需實作 `Recv BatchRequest -> callback -> Send BatchResponse`，協定簡潔且能節省 unary header 成本。

### 8.2 Remote callback vs remote-owned dispatch
- **選擇 callback adapter**：`pkg/remote` 不持有 actor dispatcher、ownership checker、worker pool 或 response coalescing policy。
- **理由**：dispatch 保序、batch 拆分與 response batching 都是 Node runtime 策略，不是 transport contract。
- **結論**：`pkg/remote` 只提供 `BatchHandler` 與 `BatchSender`。

### 8.3 Node-Scoped Batch vs Server-Side Regroup
- **選擇 Node-Scoped Batch**：stream peer 送來的 batch 應該已經是給當前 Node 的資料。
- **拒絕 Server-Side Regroup**：收到跨 Node batch 後再拆分或轉送，會讓 Node 變成隱性 gateway，破壞 No Internal Hops，也讓錯誤歸因與 backpressure 變模糊。
- **結論**：送錯 Node 的 envelope 由 Node runtime 回 `ERR_WRONG_NODE`。

## 9. 實作切分建議 (Build Sequence)

1. 建立 `pkg/remote/proto/actor_cluster.proto`。
2. 建立 `pb` 生成流程，產出 `pkg/remote/pb/*.pb.go`。
3. 完成 `remote.Server`、`BatchHandler`、`BatchSender`、error code。
4. 完成 `StreamMessages` 的 Recv / callback / serialized Send 流程。
5. 建立 `cmd/node/main.go`，掛載 `grpc.Server` 與 `pb.RegisterActorServiceServer`。
6. 後續在 Node runtime 補齊 ownership、deadline、wrong-node、payload size、op_code 驗證與 response batching。
7. 最後做 benchmark：
   - single node single stream
   - single node multi stream
   - 1 / 8 / 64 / 256 batch size
   - wrong-node fast reject
   - large payload reject

## 10. 避坑紀錄 (Pitfalls & Workarounds)

- **不要只回 `tx_id` 而省略 `request_id`**：`tx_id` 是業務冪等鍵，不是 transport correlation id；`RemoteResult` 必須帶回 `request_id`。`tenant_id` / `uid` / `op_code` 由 `completeResult` 自動回填，手動建立的錯誤 result 應明確設定這三個欄位以利 audit。
- **不要在 Node server 端自動重送 uncertain request**：Node 只回覆處理結果，不做外層重試策略。
- **不要接受錯誤 Node 後幫忙內部轉送**：一旦這樣做，cluster 內部就等於偷偷長出第二套 protocol。
- **不要在 Server 端替跨 Node batch 重新分組**：送錯 Node 的 envelope 由 Node runtime 拒絕。
- **不要把 actor dispatch 策略塞回 `pkg/remote`**：`remote` 是 transport adapter，worker pool、同 Actor 保序與 response coalescing 都屬於 Node runtime。
